# Part B — Conversation Scraping Playbook (Train Amigo + Toon on Laudi's Tone)

> Goal: build a high-fidelity training corpus from Laudi's real Airbnb inbox so Amigo (sales/diplomacy) and Toon (service/operations) speak exactly like him — same warmth, same pacing, same sales tactics, same diplomacy when things go sideways.

### Why Airbnb's API won't give us this
Airbnb does **not** offer a public messaging API to extract host inbox conversations. The only legitimate routes are:
1. **GDPR data export** ("Request your personal data") from Airbnb — slow, broad, but legal and complete
2. **Browser-session scraping** of `https://www.airbnb.com/inbox` using a logged-in headless session
3. **Channel Manager integrations** (Hospitable, Hostex, Smartbnb, Hostaway) — they have read access to Airbnb conversations via authorised partner APIs

### Recommended stack

**Step 1 — Legal & ethical baseline**
- Run a GDPR data download from Airbnb under Laudi's account (Account → Privacy & sharing → Request your personal data). Includes message archive, reviews, guest profiles. Takes 24–72 h. This is the cleanest starting corpus and is bulletproof legally.
- For ongoing capture, decide between (a) **Hospitable** (the cleanest channel-manager API for inbox + AI replies — has its own AI you can compete with by training on Laudi's voice) or (b) a custom Playwright/Browserbase scraper.

**Step 2 — Browser-based capture (if going custom)**
- **Browserbase** (serverless Chromium with persistent profiles) or **Playwright** running on a small VPS
- Persisted Airbnb login via cookies stored in a vault (1Password Connect, AWS Secrets Manager)
- Daily delta sync: poll `/inbox/messages` and `/users/show/{guest_id}` JSON endpoints (these return JSON when the right headers are set)
- Store raw HTML + parsed JSON + plain-text transcript per thread in a `conversations/` directory in the houseofwander repo (consider Git LFS or an S3 bucket — these get big fast)
- **Rate-limit:** Airbnb is sensitive — keep < 1 req/2s, randomise jitter, rotate user-agent, run from a Belgian residential IP if possible

**Step 3 — Schema for the training corpus**

```yaml
thread_id: 1234567890
listing_slug: casa-cabane
guest:
  first_name: "Marie"
  country: "FR"
  trip_purpose: "couples-weekend"
  party_size: 2
  pets: false
  age_bracket: "30-40"
booking_status: confirmed | inquiry | declined | cancelled | post-stay
turns:
  - sender: guest
    timestamp: 2025-08-01T14:23Z
    text: "Hi Laurens, is the property available...?"
    intent: availability_check
  - sender: host (Laurens)
    timestamp: 2025-08-01T14:26Z
    text: "Hi Marie, ..."
    tactics_used: [warm_open, micro-personalisation, anchor_to_lake_view, soft_upsell_outdoor_bath]
    response_time_minutes: 3
outcome:
  booked: true
  total_nights: 3
  ADR_eur: 245
  guest_review_rating: 5
  guest_review_text: "..."
  host_review_text: "..."
```

**Step 4 — Labelling pipeline (the 80/20 of agent quality)**
Run each turn through a Claude/GPT labeller that tags:
- **Intent:** availability, price, late-checkout, refund, complaint, post-stay-thanks, neighbour-tip-request, etc.
- **Tactic:** warm_open, mirror_language, anchor_value, scarcity, soft_close, give-before-ask, empathy-first, name_a_neighbour_business
- **Tone vector:** formality 0-1, warmth 0-1, brevity 0-1, humour 0-1
- **Outcome:** booked / lost / escalated / 5-star follow-up
- **Diplomacy moves** (for complaints): acknowledge → take ownership → offer concrete fix → invite continued relationship

The resulting JSONL becomes:
1. The **few-shot library** for Amigo's RAG
2. The **fine-tune dataset** if you ever want a Laudi-tuned model (300+ labelled threads is enough for LoRA on a 7B; 1000+ for something portable)
3. The **eval suite** — hold out 50 threads, see if Amigo's reply matches Laudi's in tone & outcome

**Step 5 — Two agents, two specialisations**

| Agent | Surface | Trained on | Auto-send? |
|---|---|---|---|
| **Amigo** (sales/concierge) | Pre-booking inquiries, upsells, neighbourhood questions, late-night reassurance | Inquiry threads + 5-star post-stay reviews | **No** — drafts only, Laudi hits send |
| **Toon** (operations/service) | Check-in instructions, lost items, broken-thing complaints, cleaning issues, refund negotiations | Confirmed-booking through post-stay threads, complaint resolutions | **No initially** — full draft, Laudi approves; **Yes later** for templated intents like "wifi password" |

Both share a **forbidden actions** list (never quote refunds without Laudi approval, never make exceptions to the cancellation policy, never promise late check-out without checking the cleaning calendar).

### Voice fingerprint — what to extract from Laudi's existing messages

Have the labeller surface these explicit patterns so the system prompt can lock them in:
- **Opening signatures** ("Hi Marie!" vs "Hey Marie" vs "Dear Marie") — frequency map
- **Sign-offs** ("Warm regards", "Until soon", "Cheers from Ghent")
- **Recurring micro-phrases** ("our little hidden retreat", "if there's anything at all", "we'd love that")
- **Emoji usage** (heatmap by intent — usually never in complaint replies, often in pre-booking)
- **Language switching policy** (when does Laudi switch NL ↔ FR ↔ EN ↔ DE ↔ ES?)
- **Average response length per intent**

### What Toon and Amigo should each NEVER do (initial guardrails)
- Never make up an amenity not in the listing
- Never invent a discount
- Never reveal exact address before booking confirmation
- Never agree to off-platform payment (Airbnb ToS violation that kills Superhost status)
- Always escalate: legal threats, injury/safety, "I want a refund and I'm contacting Airbnb"

# Part C — Product / Design Brief for House of Wander

> Based on the current `ArtificialMonks/houseofwander` repo (Next.js 15 + TypeScript, app/ + source/ + public/media + docs/, with a Three.js gateway, a Casa Cabane walkthrough video, and a scripted Amigo chatbot prototype that labels source boundaries) and the listings we just scraped.

## C.1 — What's already strong (don't break it)

1. **Source-boundary labelling in Amigo.** This is the right move. Most travel chatbots invent. Yours admits what it knows. Keep this *visibly* in the UI — it's a differentiator vs. Airbnb itself.
2. **3D brand gateway.** Distinctive and on-brand for "wander". Just make sure it lazy-loads gracefully on mobile.
3. **`source/` directory as source of truth.** Excellent architectural call. The knowledge base above is designed to drop straight into it as `source/listings/*.json` or `*.md`.
4. **Versioned renders** (`renders/casa-cabane-poster/20260523-042046-v01/`). Treat all hero media this way.

## C.2 — High-impact features to add (ranked by ROI)

### 1. Direct-book layer that beats Airbnb on every dimension
- **Why:** Airbnb takes 14–16% in combined fees. On €100/night avg × ~30 nights/property/month × 7 properties × 12 months = **~€60–80k/year in fees you can recapture** by even 30% direct-book shift.
- **How:**
  - Stripe Connect + Bokun/Octorate/Smoobu calendar sync (so Airbnb doesn't get a double-booking)
  - Lower price by 5% vs Airbnb but pocket 9% — guest wins, you win
  - "Book direct, get a bottle of local wine on arrival" — costs €8, converts 2× harder
- **Track:** `direct_book_rate` per property, target 25% by month 6

### 2. Heritage Collection landing pages (5 unbuilt routes)
Add `/stays/heritage/[child-name]` routes for the 5 unmapped Heritage Collection apartments. Each gets the same walkthrough treatment as Casa Cabane. Crucial — the collection narrative *requires* all five visible.

### 3. Amigo → real agent (not scripted)
Right now Amigo is a scripted UI prototype. Replace with:
- **Claude Sonnet 4.6 or Haiku 4.5** behind the chat (Haiku for speed/cost, Sonnet for complex objection-handling)
- **RAG over `source/listings/` + the conversation corpus from Part B**
- **Tool calls:** `check_availability(slug, dates)`, `get_quote(slug, dates, guests)`, `create_hold(slug, dates)`, `escalate_to_laurens(thread_id, reason)`
- **Always-visible "Why I know this" panel** — shows the exact source snippet Amigo's pulling from
- **Multilingual** — NL / FR / EN / DE / ES (match Laudi's languages)

### 4. Trust shelf — the "social-proof spine" of every stay page
Pull from the public Airbnb data we scraped:
- Rating badge (★ 4.96 · 155 reviews) with link to Airbnb reviews
- "Guest favourite" / "Top 5%" badges where applicable
- 3–4 rotating real review snippets per page (with permission)
- Superhost badge with Laurens' face + "5 years hosting · 100% response · within 1 hour"
- "What guests notice most" word cloud from the review topics array

### 7. Multi-property cart
Travellers planning Belgium often want Ghent + coast. Let them book Casa Fabiola + Louise Marie together with one checkout, one discount ("Wander Pass: 10% off when you book 2+ stays").

### 9. Cross-platform parity table on every stay page
Show "Book here / Book on Airbnb / Book on Booking.com" — transparent, builds trust, captures direct-bookers who'd otherwise comparison-shop.

### 10. Mobile-first refresh
Currently the README says desktop/phone/tablet were checked — confirm the 3D gateway, the Casa Cabane video, and the Amigo chat all degrade gracefully under 4G. Lighthouse mobile score target: 90+ on Performance, 100 on SEO/Accessibility.

- **Logo lockup** — the "House of Wander" wordmark needs a permanent place in the header + an animated reveal on the gateway
- **Property naming hierarchy** — every listing should display *both* the marketing name (Casa Cabane) and the public Airbnb title ("Stylish lakeside house, green nature"). Currently the Airbnb titles are flat and feel like SEO bait; the marketing names are the brand asset.
- **Photography pass** — the existing photos are good (especially Love Nest and Casa Fabiola); commission a unified shoot in autumn 2026 for tonal consistency (warm naturals, real people, candid moments — match Airbnb 2025 aesthetic but with more "still" / less staged)
